These labs aim to be a practical scenario on how to use network tools such as **nmap** and **netcat** perform a network enumeration without being too noisy. In these labs, the following nmap flags are **mandatory** if you want your scan to be as quite as possible (and also faster):

|flag|purpose
|-|-
|-Pn|disables ICMP host discovery
|-n|disables DNS resolution making the scan faster
|--disable-arp-ping|disables ARP ping host discovery 
|-sS (default when running nmap as sudo)|perofrms sealth scan 
|-D|hides the scan ip address making the scan less discoverable
|--max-retries 0|makes the scan faster but at the cost of precision
 
## Easy Evasion Lab

- **task:** Our client wants to know if we can identify which operating system their provided machine is running on. Submit the OS name as the answer. 

There are several ways to solve this excercise without being too noisy on the network, even without using nmap. From the excercise we know the machine is serving a website to show the number of alerts we have caused in the system, since port 80 is open, we can try to analize if the **server** http header gives us valuable information:

### Getting the Answer with the Server http Header

```bash
[haxx0@archlinux ~]$ curl -I 10.129.109.183/status.php
HTTP/1.1 200 OK
Date: Wed, 18 Dec 2024 19:43:23 GMT
Server: Apache/2.4.29 (Ubuntu)
Content-Type: text/html; charset=UTF-8
```

From the HTTP headers we know the OS turns out to be **Ubuntu**

### Getting the Answer with netcat and nmap

From what the excercise says, we only know that an **IDS/IPS** system is being implemented, no firewall at the moment, but let's assume a firewall is present. We want to determine the operating system of the machine, one port that is almost guaranteed to give us the answer is the port 22. We can try to perform manual banner grabbing with netcat to avoid scanning the machine. We can also scan the machine with nmap and perform some tricks to avoid being detected:

```bash
[haxx0@archlinux ~]$ nc 10.129.109.183 22
SSH-2.0-OpenSSH_7.6p1 Ubuntu-4ubuntu0.7
```

![amount of traffic generated by netcat](images/image.png)

I performed the following scan with nmap, it ended up being a bit more noisy than netcat, still not very discoverable thanks to the decoys.

```bash
[haxx0@archlinux ~]$ sudo nmap -Pn -n --disable-arp-ping -p22 --max-retries 0 -sV -D RND:5 10.129.109.183
```

## Hard Evasion Lab

**task:** Now our client wants to know if it is possible to find out the version of the running services. Identify the version of service our client was talking about and submit the flag as the answer. 

From what we are told, the client implemented a new service, we know ports 22 and 80 are discoverable so it must be a different port. We can also assume this service has to be reachable so the port is not going to be filtered, still, we might want to be careful with the scan since we know there is an **IDS/IPS** system, so I will still apply the same meassures I have been applying:

```bash 
[haxx0@archlinux ~]$ sudo nmap -Pn -n --disable-arp-ping -D RND:5 -g 53 10.129.249.42 --max-retries 0
Starting Nmap 7.95 ( https://nmap.org ) at 2024-12-18 14:51 CST
Warning: 10.129.249.42 giving up on port because retransmission cap hit (0).
Nmap scan report for 10.129.249.42
Host is up (0.071s latency).
Not shown: 869 closed tcp ports (reset), 129 filtered tcp ports (no-response)
PORT      STATE SERVICE
22/tcp    open  ssh
50000/tcp open  ibm-db2

Nmap done: 1 IP address (1 host up) scanned in 2.53 seconds
```

Port **50000** is a new port, we can try to get its banner with netcat. I initially did the scan without specifying port 53 as my source port, without that, the port will be ignored and will appear as **filtered**, thus, when performing banner grabbing we should also specify port 53 as the source of the scan:

```bash
[haxx0@archlinux ~]$ sudo nc -nvp 53 10.129.249.42 50000
10.129.249.42 50000 open
220 HTB{kjnsdf2n982n1827eh76238s98di1w6}
```

And there we have it! All the nmap labs have been successfully solved.
